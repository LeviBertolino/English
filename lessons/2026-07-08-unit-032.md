## 2. 🧠 Adquirir — Mini-context first, rule second

> **Levi, staring at a monitoring dashboard with a teammate:**
>
> **Teammate:** CPU usage jumped to 100% at 3am and just stayed there.
> **Levi:** That **must be** the nightly batch job — it always spikes memory around that time.
> **Teammate:** But the error logs are completely empty before the crash.
> **Levi:** Then it **can't be** a code bug — it **must be** something external, like a noisy neighbor on the shared cluster.
> **Teammate:** Makes sense. I'll check the infra team's deploy history for 3am.

Notice:
- **must + base verb** = "I'm almost certain this is true" — a strong logical conclusion, not permission or obligation.
- **can't + base verb** = "I'm almost certain this is NOT true" — the negative twin of *must*, based on evidence.
- Neither sentence is a guess like *may/might* (Units 30–31) — *must* and *can't* express near-**certainty**, reasoned from evidence in front of you.

### The rule

| Form | Use | Example |
|---|---|---|
| **must** + base verb | certain something is true (now) | She's not answering — she **must be** in a client call. |
| **can't** + base verb | certain something is **not** true (now) | The API **can't be** down — I just called it successfully. |
| **must have** + V3 | certain conclusion about the **past** | The deploy **must have broken** something — errors started right after. |
| **can't have** + V3 | certain something did **not** happen | He **can't have read** the RFC — his comment contradicts page one. |

> ⚠️ Common trap: for negative certainty, English uses **can't** (or **couldn't**), never *mustn't*. ❌ *"It mustn't be a caching issue"* → ✅ *"It **can't be** a caching issue."*
> ⚠️ Don't confuse this *must* (deduction) with the *must* of obligation ("You must submit the report by Friday") — same word, very different job. Context tells you which one.
> ⚠️ *Must have* vs *might have*: *must have* = the evidence leaves little doubt; *might have* = one possibility among others.

## 3. ✏️ 8 example sentences

1. The deploy **must have broken** something — errors started right after it shipped.
2. She's not answering Slack; she **must be** in a client call.
3. This **can't be right** — the dashboard shows negative latency.
4. The on-call engineer **must be** exhausted; she's been paging all night.
5. He left the porch light on all night; he **must have forgotten** to check before bed.
6. The neighbor's dog hasn't barked all morning; it **must be** sleeping.
7. The restaurant is closed on a Tuesday afternoon; it **can't be** that popular anymore.
8. My phone died within an hour of a full charge; it **must have** a defective battery.

## 5. 💡 Memory tip

**MUST = 💡 (I'm sure — yes). CAN'T = 🚫 (I'm sure — no).** There's no "mustn't" in this game — when you want to kill a theory with confidence, only **can't/couldn't** does the job.

## 8. 🇧🇷 Resumo em português

Hoje vimos **must** e **can't** usados para **dedução/certeza**, não para obrigação ou proibição. **Must + verbo** expressa uma conclusão quase certa de que algo **é verdade** ("deve ser..."); **can't + verbo** expressa uma conclusão quase certa de que algo **não é verdade** ("não pode ser..."). Para o passado, usamos **must have + particípio** e **can't have + particípio**.

O erro mais comum de falantes de português é tentar usar **mustn't** para negar uma certeza — em inglês isso não existe nesse sentido; **mustn't** é só proibição ("você não deve fazer isso"). Para dizer "tenho certeza que não", a forma correta é sempre **can't** ou **couldn't**. Cuidado também para não confundir esse *must* de dedução com o *must* de obrigação (Unit 33, em breve) — a palavra é a mesma, mas o sentido depende totalmente do contexto.
